## 2. 專案背景

PO 希望在奇摩購物網站中加入「AI 推薦」功能，例如根據使用者行為推薦商品，但團隊目前沒有相關經驗。

本專案是 Full-Stack / Web 全端工程師面試前測。重點不是打造正式可上線的推薦系統，而是在有限時間內展現：

1. 如何拆解產品需求。
2. 如何評估 AI 推薦功能的導入方式。
3. 如何使用 AI 輔助工具完成可展示的 MVP。
4. 如何說明技術取捨、限制與後續演進。

---

## 3. 專案目標

### 核心目標

建立一個「奇摩購物 AI 推薦功能 MVP」，讓使用者在瀏覽商品或操作行為後，可以看到一組看似個人化的推薦商品。

### 產品目標

1. 提升商品曝光與導購效率。
2. 讓使用者更容易發現可能感興趣的商品。
3. 先用低成本方式驗證推薦區塊是否有價值。
4. 為未來接入真實 AI / ML 推薦系統預留架構。

### 技術目標

1. 以可展示的前端 MVP 為優先。
2. 推薦邏輯先採用「規則式 + mock 使用者行為」模擬 AI 推薦。
3. 程式結構需保留未來替換成第三方推薦服務或自建模型 API 的空間。
4. 使用 AI 工具協作產出需求、原型、程式與說明文件。

## 4. 實作策略

本專案採用「先 MVP、後 AI 化」策略。

### Phase 1：規則式 MVP（本次實作）

- 使用 mock 商品資料。
- 記錄或模擬使用者行為，例如瀏覽商品、點擊分類、加入購物車、搜尋關鍵字。
- 根據行為權重推薦商品。
- 顯示推薦理由，讓面試官能理解推薦結果為何出現。

### Phase 2：第三方 AI / 推薦服務（未來規劃）

- 評估 SaaS 推薦服務、向量資料庫或 LLM 輔助商品理解。
- 將推薦邏輯封裝為 `recommendationService`。
- 前端只依賴統一介面，不直接綁定推薦演算法。

### Phase 3：自建模型（長期規劃）

- 累積足夠行為資料後，再評估 collaborative filtering、content-based filtering 或 hybrid recommendation。
- 補齊資料管線、特徵工程、模型評估與 A/B test 機制。

2. 使用者行為模擬或簡易記錄。

## 6. 使用者情境

### 主要角色

- 一般購物使用者。
- 產品負責人 / 面試官。
- 開發者。

### 核心流程

1. 使用者進入購物頁面。
2. 系統顯示商品列表與預設推薦商品。
3. 使用者瀏覽、點擊或加入商品。
4. 系統更新使用者行為狀態。
5. AI 推薦區塊重新排序推薦商品。
6. 使用者可看到推薦理由。

## 7. 功能需求

### FR-001 商品資料展示

系統需顯示 mock 商品資料，至少包含商品名稱、分類、品牌、價格、圖片、熱度或評分。

### FR-002 使用者行為追蹤

當使用者點擊商品、瀏覽商品、搜尋關鍵字或加入購物車時，系統需記錄該行為並更新推薦依據。

### FR-003 AI 推薦區塊

系統需顯示「AI 為你推薦」區塊，依使用者行為即時更新推薦商品。

### FR-004 推薦理由

每個推薦商品需顯示簡短推薦理由，例如：

- 因為你瀏覽過相同分類。
- 與你剛剛查看的商品價格相近。
- 同品牌熱門商品。
- 根據你的搜尋關鍵字推薦。

### FR-005 推薦邏輯封裝

推薦計算需封裝為獨立服務或函式，避免散落在 UI 元件中，方便未來替換為 API 或第三方服務。

### FR-006 空狀態處理

若尚無使用者行為，系統需顯示預設熱門推薦。

---

## 8. 非功能需求

### NFR-001 可展示性

專案需能在本機快速啟動並展示完整流程。

### NFR-002 可理解性

程式與文件需清楚表達「目前是 MVP 模擬推薦，不是真實模型」。

### NFR-003 可擴充性

推薦資料來源需可由 mock function 替換為後端 API。

### NFR-004 前端體驗

畫面需具備購物網站語境，推薦區塊需清楚、可掃讀、可互動。

### NFR-005 可維護性

商品資料、使用者行為、推薦服務與 UI 元件需分離，避免單一檔案承擔過多責任。

## 9. 推薦邏輯規定

本次推薦演算法使用規則式 scoring。每個候選商品可依下列條件計分：

| 條件 | 建議分數 |
|:--|:--|
| 與最近瀏覽商品同分類 | +40 |
| 與最近瀏覽商品同品牌 | +25 |
| 價格區間相近 | +15 |
| 商品熱度高 | +10 |
| 使用者搜尋關鍵字命中 | +20 |
| 已瀏覽商品 | 排除或大幅降權 |

實作時需輸出推薦理由，且推薦理由需能對應到 scoring 條件。

## 10. 架構與檔案規則

### AGENT.md 分層關係

本專案採用「根規範 + 子專案規範」結構：

| 檔案 | 適用範圍 | 用途 |
|:--|:--|:--|
| `AGENT.md` | 全專案 | 定義產品目標、MVP 範圍、推薦策略、交付物與驗收標準 |
| `frontend/AGENT.md` | 前端 | 定義 Next.js + TypeScript 的 UI、資料流、API client 與檔案規則 |
| `backend/AGENT.md` | 後端 | 定義 C# .NET Core RESTful API 與 Controller / Service / Repository 分層規則 |

當任務只修改前端時，需同時遵守 `AGENT.md` 與 `frontend/AGENT.md`。

當任務只修改後端時，需同時遵守 `AGENT.md` 與 `backend/AGENT.md`。

當任務跨前後端時，需同時遵守三份規範，並確保 API contract 一致。

### 技術棧規定

- 前端必須使用 Next.js + TypeScript。
- 後端必須使用 C# .NET Core RESTful API。
- 後端架構必須採用 Controller / Service / Repository 分層設計。
- 若專案尚未初始化，先建立最小可執行的前後端專案骨架。
- 資料先使用 mock data。
- MVP 階段可使用 in-memory repository 或 mock repository，但仍需保留 Repository 介面與分層。
- 不強制接真實 AI API，除非使用者明確要求。

### 建議 RESTful API

| Method | Endpoint | 用途 |
|:--|:--|:--|
| `GET` | `/api/products` | 取得商品列表 |
| `GET` | `/api/products/{id}` | 取得單一商品 |
| `POST` | `/api/recommendations` | 依使用者行為取得推薦商品 |

### 推薦功能至少拆分為

1. 商品資料：`mockProducts` 或 `products`。
2. 使用者行為狀態：`userEvents` 或 `behaviorState`。
3. 後端推薦服務：`IRecommendationService` / `RecommendationService`。
4. 後端商品資料存取：`IProductRepository` / `InMemoryProductRepository`。
5. 前端 API 呼叫：`recommendationApi` 或 `apiClient`。
6. 前端 UI 區塊：`RecommendationSection`。

### 專案根目錄建議結構

```text
frontend/
├── AGENT.md
backend/
├── AGENT.md
docs/
README.md
AGENT.md
```

## 11. UI/UX 規則

- 第一畫面必須直接是可操作的購物推薦體驗，不做行銷 landing page。
- 商品卡片需包含商品名稱、分類、品牌、價格與可互動按鈕。
- 推薦區塊需明確標示「AI 為你推薦」或類似文案。
- 推薦理由要短、清楚、可掃讀。
- 使用者操作後需有明確狀態變化，例如最近瀏覽、已加入購物車、推薦結果重排。
- 介面需支援桌機與手機基本響應式。
- 不要使用過度裝飾的頁面；以購物網站、商品瀏覽、推薦展示為主。

## 12. 驗收標準

### AC-001

Given 使用者第一次進入頁面，When 尚未有任何互動，Then 系統顯示預設熱門推薦商品。

### AC-002

Given 使用者點擊某個商品，When 行為被記錄，Then 推薦區塊需更新為同分類、同品牌或相近價格商品。

### AC-003

Given 推薦商品被顯示，When 使用者查看推薦區塊，Then 每個商品需顯示推薦理由。

### AC-004

Given 開發者查看程式碼，When 尋找推薦邏輯，Then 可以在獨立的推薦服務或函式中找到核心 scoring 規則。

### AC-005

Given 面試官觀看展示，When 詢問是否為真實 AI 模型，Then 專案文件需清楚說明目前為 MVP 模擬，並提供後續導入真實 AI 的路線。

## 15. 開發工作規則

- 修改程式前先理解現有檔案結構。
- 優先沿用現有專案技術棧與風格。
- 若尚未有專案架構，建立最小可執行版本。
- 推薦邏輯需放在後端 Service，並可單獨閱讀與測試。
- 不要把 mock data 寫死在 UI 元件中。
- 前端不得直接實作後端商業邏輯；前端只負責呈現、互動與呼叫 API。
- 後端 Controller 不得直接存取資料來源；必須透過 Service。
- 後端 Service 不得直接依賴具體 Repository 實作；必須依賴 Repository interface。
- 不要為了面試題過度加入後端、資料庫、登入、付款等非必要功能。
- 完成後需至少執行前端 build 與後端 build，或其他可用的驗證指令。
- 若無法執行測試或 build，需在回覆中說明原因。
